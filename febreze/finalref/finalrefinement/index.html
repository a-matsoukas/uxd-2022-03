Final Refinement Phase Submission

Modifying Design & Implementing Changes

In the design refinement stage, our modification process was driven by an overarching goal for our interface that we came up with as a team. We wanted our interface to be very smooth and efficient for the user to navigate through. With this in mind, we started modifying our interface by taking the feedback we got from our heuristic evaluation and making all necessary changes that hindered the usability of our interface. We then navigated through our interface multiple times, and fixed minor issues we found in testing and in our feedback. After arriving upon two iterations we were happy with, we began our A/B testing with users who had not previously seen our interface. We recorded very helpful data through our testing, and used this data to compare the two prototypes and properly refine our interface.

Addressing Heuristic Violations (Level 3+)

From the heuristic evaluations, we had two violations of Severity 4, and four violations of Severity 3.

Navigating to the Business Page from the Photo Carousel [#1: Visibility of System Status] (Severity: 4)

After clicking on the Business Page, you are taken to a Photo Carousel. While the indicator on the bottom shows that there are multiple pictures to view, it does not indicate that you should swipe up for more information. Our group got caught up swiping in circles, unsure of how to advance. Adding an indicator to swipe up would make this easier to use.

This was was one of the features that we tested with in our usability study, where we performed A/B testing with two new versions of a swipe indicator: one with a small up arrow, and the other with a bigger hand icon, as shown below:

Our users were asked to find the “Directions” button, after which we timed how long it took for the user to realize they needed to swipe up from the location snapshot to view its details, and measured how many taps/interactions it took for them to get to the details page by eventually swiping up. We thought that it would be slightly more obvious that the user needs to swipe up with the larger indicator, making it faster for users to find this function, and by the end of the experiment, we were correct in this assumption. The old version of the prototype did not contain any sort of icon or indicator to swipe up, as brought up in this heuristic violation, so it was a bit unintuitive for users to figure out how to view the details of a certain location from the snapshot, hence why we decided to test this feature.

Potential for conflicting input [#5 Error Prevention] (Severity: 4)

When entering information during the sign up process there is the opportunity to enter conflicting information about your requirements (i.e. on the “Do you need to avoid certain surfaces?” page the user can select both “no” and other options. This could confuse the algorithm on the requirements that need to be met when selecting options. Adding some kind of preventative measure would make logical sense for this process.

We fixed this by simplifying questions and adding a skip button users can choose instead of saying “no” or “none” in order to add preventative measures for conflicting input. This was another feature that we tested in our usability study, where we performed A/B testing with two versions of the personalization quiz: the old version without the skip button and progress bar, and the new version with both of these added. As expected, the version with these added features was preferred.

“Saved” page doesn’t have direct access to the Map [#7 Flexibility and efficiency of use] (Severity: 3)

Users don’t have direct access to the map from the “saved” page which goes against the pattern recognition of the map being easily accessed from any page in the app.

We added a Map button on the Saved page for the user to be able to directly access the map from the Saved page without having to go back to another page in order to access it.

You are here indicator on the Map [#1 Visibility of System Status] (Severity: 3)

Including a marker for users to know where they are relative to a location on a map would help them have a better understanding of where they are and how to get there.

We added a pinpoint indicator on the map so that users are able to know where they currently are on the map, in order to facilitate their traveling.

Linking between Krusty Krab Back Button and Map Page [#1 Visibility of System Status, #3 User Control and Freedom] (Severity: 3)

After clicking on the Krusty Krab business page from the Home page, we tried to exit using the back button. Instead of being taken back to the Home page as expected, we were taken to the Map page. This design feature made it feel like the interface had imposed an action on the user and had decided that we should continue looking for places in the area.

This was fixed such that when users click on a business page from the home page and then hit the back button, they are routed back to the home page rather than the map.

Hidden ratings [#4 Consistency and Standards] (Severity: 3)

Highlight the rating a bit more on the page since users will care about rating most after accessibility concerns.

We decided to not fix the above issue because this was very similar to another one that we had fixed before, related to ratings and aesthetics. Hence, we believe we tackled this problem already.

We also decided not to add a feature that would show directions to the user. Instead, we added the option to view any location selected in a separate map app. While we think this feature could have added value to our app, we decided it was out of the scope of this project. There were also limitations to what could be quickly implemented in Figma, such as the ability to zoom in and out on a map. Although it would have been nice to include every feature we wanted to, it was important to prioritize getting usability feedback before overcomplicating the interface.

There were also some minor aesthetic changes that we decided not to pursue because we did not agree with the feedback team.
